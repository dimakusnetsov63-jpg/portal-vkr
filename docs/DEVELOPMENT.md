# Разработка

Практический workflow для портала ВКР. Правила и архитектура —
[`../CLAUDE.md`](../CLAUDE.md) и [`ARCHITECTURE.md`](ARCHITECTURE.md).

## Запуск проекта

```bash
npm install        # установка зависимостей
npm run dev        # dev-сервер (next dev)
```

По умолчанию dev-сервер поднимается на `http://localhost:3000`.

## Переменные окружения

Создать `.env.local` в корне (файл в `.gitignore`, не коммитить):

```
NEXT_PUBLIC_SUPABASE_URL=<url проекта Supabase>
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=<publishable ключ>
```

Значения — из Supabase Dashboard проекта. Реальные ключи в чат/документацию/
git не попадают. На Vercel те же переменные заданы в настройках проекта.

Без этих переменных клиент Supabase бросит понятную ошибку
(`Missing environment variable…`, см. `lib/supabase/env.ts`).

## Проверки

```bash
npx tsc --noEmit   # проверка типов (отдельного npm-скрипта нет)
npm run lint       # ESLint (eslint-config-next)
npm run build      # production-сборка (next build)
```

- `tsc` проверяет **только типы** — не логику и не UI.
- `lint` не заменяет функциональную проверку.
- `build` компилирует и проверяет типы, но не гарантирует корректность
  интерфейса.
- **UI-изменения проверяются в браузере.** Раздел «Кандидаты» и большинство
  разделов — за авторизацией; для проверки нужен тестовый сотрудник, заведённый
  в Supabase Dashboard.

## Структура новой функциональности

Держите код по ответственности:

- **Оркестратор раздела** (`*Section.tsx`) — состояние, данные из контекста,
  композиция.
- **Подкомпоненты** (таблицы, модалки, drawer) — отдельные файлы в папке раздела.
- **Чистая логика** (фильтры, метрики, преобразования) — `.ts`-файлы без React
  и Supabase.
- **Доступ к БД** — только через репозитории `lib/supabase/*Repo.ts`, вызываемые
  из `PortalContext`.

## Как добавить новый раздел

1. `sections/<name>/<Name>Section.tsx` — оркестратор.
2. Подкомпоненты — отдельными файлами в той же папке; чистая логика — в `.ts`.
3. Данные — через `PortalContext` (не звать Supabase из компонента напрямую).
4. Пункт меню — в `NAV_ITEMS` (`lib/portal/constants.ts`); ветку рендера — в
   `ActiveSection` (`PortalApp.tsx`).
5. Переиспользовать примитивы из `ui/`.

## Как добавить поле кандидата

1. Миграция в `supabase/migrations/` (новое поле, при необходимости индекс,
   дефолт, nullable).
2. Применить миграцию и **регенерировать** `database.types.ts`
   (`supabase gen types typescript`) — вручную типы не править.
3. Поле автоматически появится в `Candidate`/`CandidateInsert`/`CandidateUpdate`.
4. Обновить UI: `RealCandidateDrawer` (редактирование), при необходимости
   `AddCandidateModal`, `CandidatesTable`, CSV-экспорт, фильтры/метрики.
5. `tsc` → `lint` → `build`; проверить в браузере.

## Как менять Supabase-схему

- Только через новую миграцию — существующие применённые миграции не править.
- После изменения — регенерация типов.
- Названия существующих полей и enum не менять без отдельного согласования.
- Значения проектов/стадий заданы бизнесом дословно — не переименовывать.
- Подробнее — [`DATABASE.md`](DATABASE.md).

## Безопасный рефакторинг

- Двигаться маленькими логическими этапами.
- Переносить код **дословно**, отдельно от смысловых изменений.
- Для перемещений использовать `git mv` (сохраняет историю).
- После каждого этапа — `npx tsc --noEmit`.
- В конце — `lint` + `build`.
- Не менять публичный API компонентов и `usePortal()` без необходимости.
- Перед удалением файла — grep всех использований по проекту.

Пример уже выполненного безопасного рефакторинга — раздел «Кандидаты»
(коммит `76616ed`): перемещение в папку + вынос чистой логики + разбиение
компонента, без изменения поведения.

## Коммиты

- Одна логическая задача — один коммит.
- Не смешивать рефакторинг, брендинг и функциональные изменения.
- Перед коммитом: `git diff --check`, `git status --short`.
- Не коммитить `.env.local` и любые секреты.
- Сообщения по смыслу: `type(scope): summary`
  (напр. `refactor(candidates): split section into modules`).

## Проверка UI после изменений

1. `npm run dev`, открыть затронутый экран.
2. Проверить консоль браузера и сетевые запросы на ошибки.
3. Для экранов за авторизацией — войти тестовым сотрудником.
4. Проверить адаптивность/тёмную тему, если менялась вёрстка.
5. Не заявлять «работает», не увидев результат.

## Чего нельзя делать без отдельной команды пользователя

- `git push`, deploy.
- Применять миграции к боевой БД.
- Destructive-команды (`git reset --hard`, `git clean`, удаление файлов/данных).
- Менять дизайн, зависимости, конфигурацию, Supabase-схему «заодно».
- Использовать `service_role`/секретные ключи в клиентском коде (запрещено
  всегда).
